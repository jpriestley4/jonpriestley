Project Overview

The brief

The goal of this project was to conduct a WCAG 2.1 accessibility audit of mycarebudget.org up to an AA conformity level. Mycarebudget.org is a website which helps people manage their budgeting for care services.

Role

  • UX Designer

Duration

  • 2 Months

Skills

  • WCAG 2.1 Audit
  • HTML, CSS, JS
  • UX/UI design

3 Step Approach

1

Automated checks

To identify the most problematic and impactful accessibility issues, automated checks were performed using WAVE, a web accessibility evaluation browser plugin. This quick and dirty approach was a great way to start the audit process but a more thorough, manual review was also needed.

2

Manual checks

The manual checks involved inspecting the website's code against WCAG 2.1 guidelines and using text-to-speech software, such as Mac's VoiceOver, to test whether screenreader users can navigate the site.

3

Prioritising and reporting findings

To draw the stakeholder's attention to addressing the most serious and impactful issues, each finding was assigned a severity rating. Then, these issues were organised in a concise report covering the four main WCAG principles: Perceivable, Operable, Understandable and Robust.

Tools

WAVE Evaluation Tool

I used WAVE to automatically check the site's source code and identify any errors against WCAG 2.1 guidelines

Using WAVE helped to flag 234 potential errors across the pages being reviewed. Each error was documented in a spreadsheet to be reviewed during the manual checks.

W3C's Easy Checks

I used W3C's Easy Checks to get an overview of the most important areas to investigate for when conducting an accessibility audit.

Using Easy Checks made it easy for me to identify issues surrounding inadequate alternative text for images and heading structure.

Easy checks for accessibility:

  1. Page title
  2. Alt text
  3. Heading structure
  4. Colour contrast ratio
  5. Resize text
  6. Keyboard focus
  7. Form labels
  8. Flashing and Blinkind content

W3C's Markup Validation Service

I used W3C’s Markup Validation Service to assess whether the site’s HTML was built correctly in order for it to be compatible with various assistive technologies.

Keyboard navigation

As part of the manual checks, I navigated each page using only my keyboard. Page elements should be fully navigable using they keyboard by pressing the 'Tab' button

This ensured that each page was navigable by keyboard which is essential for users with physical impairments who aren't able to use a mouse or trackpad

Mac's VoiceOver OS Screen Reader

I used Mac's VoiceOver OS to investigate how accessible the site was with a screen reader.

The screen reader was used in two ways:

  1. Using the 'Tab' key and arrow keys to move across the page's elements chronoligically. This helps to check if the page elements are accessible via a screen reader and if the sequence of content on the page makes sense to screen reader users.
  2. Navigating the page using the heading structure menu provided on VoiceOver. This helps to check if the heading structure has a chronological sequence and that the user is able to access all content areas.

Prioritising Accessibility Issues

To help prioritise the most important accessibility issues for stakeholders to address, I modified Molich and Jeffries 3 point rating scale.

Severity Rating Fix Priority Description
Minor Low Delays user briefly.
Serious Medium Delays user significantly but eventually allows them to complete their task.
Catastrophic High Prevents user from completing their task.

Report

Report layout

The findings in the report were categorised based on the four main WCAG principles: Perceivable, Operable, Understandable and Robust.

Each finding in the report detailed the following:

  • Identified issue
  • WCAG 2.1 violation
  • Severity
  • Location
  • Issue description
  • Why it matters
  • Suggested fix

High priority issues

1. Perceivable: Information and user interface components must be presentable to users in ways they can perceive.
Issue Description

Identified issue:

Loss of content when zooming in by 400%

Violation:

1.4.10 – Reflow (Level AA)

Severity:

3 - Catastrophic

Location:

  • Home Page,
  • What’s New,
  • Just Starting Out,
  • Create Account

Identified description:

When zooming in by 400% on the ‘Home page’ and ‘Create account‘ page, the user loses content on the screen and has to scroll horizontally to find the information that was initially off the screen.

Why it matters:

Users with visual impairments may need to zoom into a page in order to perceive its content. If content becomes disorganised or lost when a user zooms into a page, then the user will miss important information that is essential for the completion of their desired task.

Suggested fix:

Use a CSS layout flexbot to define flexible layout regions on the page. This will ensure that pages do not lose content when a user zooms in to 400%.
2. Operable: User interface components and navigation must be operable.
Issue Description

Identified issue:

Inaccurate use of links.

Violation:

2.4.4 – Link Purpose (In Context) (Level A), 4.1.2 – Name, Role, Value (Level A)

Severity:

3 - Catastrophic

Location:

  • Home Page,
  • Just Starting Out,
  • What's new

Identified description:

On the Home page, Just Starting Out page and the What’s New page, the function of multiple hyperlinks within the body text of the page is ambiguous, such as ‘here, ‘click here’ or ‘what is this?’. Furthermore, there are several instances of links which share the same text label but send the user to different locations.

Why it matters:

Clear links provide a useful way for users to navigate from one page to another on a website. Visually impaired users rely on clearly defined links whilst they use screen readers to tab through the links on a page. Furthermore, clearly defined links help to prevent users with cognitive impairments from becoming disorientated on a site. In contrast, links that are ambiguous, misleading and redundant impede both visually impaired and cognitively impaired users from being able to understand the content on a page. As a result, they are unable to complete their desired task whilst on the site.

Suggested fix:

Make sure all link text clearly describes the intended function of each link. It is equally important to make sure that links which take the user to different locations, don’t share the same link text. Also, consider removing the redundant links from the home page layout table.

3. Understandable: Information and the operation of user interface must be understandable.
Issue Description

Identified issue:

Inconsistent identification of user input errors.

Violation:

3.3.1 – Error Identification (Level A)

Severity:

3 - Catastrophic

Location:

  • Create Account

Identified description:

When creating an account, the site does not identify user input errors across the input form fields: Username, Password, Confirm password, Email Address, Real Name (optional).

Why it matters:

Providing input error messages allows users who are visually impaired to identify an error via a screen reader. Furthermore, users with cognitive impairments rely on consistent error identification to help clearly identify and explain all user errors collectively.

Suggested fix:

When a user makes an error, make sure an error identification message appears next to each input field where an error has occurred.

Robust: Content must be robust enough that it can be interpreted by a wide variety of user agents, including assistive technologies.
Issue Description

Identified Issue:

Lack of aria labels used to describe user interface components.

Violation:

4.1.2 – Name, Role, Value (Level A)

Severity:

3 - Catastrophic

Location:

  • Home Page,
  • What’s New,
  • Just Starting Out,

Identified description:

On all pages, user interface components, such as links, do not have accompanying ARIA labels to describe their intended purpose to the user.

Why it matters:

For users of screen reading technology, ARIA Labels help provide an additional description of user interface components when the HTML alternative text script is not sufficient.

Suggested fix:

Ensure all user interface components that have unclear alternative text are provided with accompanying ARIA labels that describe the intended purpose of the component.
